# Oak Containers Sysroot

This directory contains the logic for building the Oak Containers sysroot using
[rules_distroless](https://github.com/google/rules_distroless).

The sysroot provides a consistent set of libraries, headers, and tools used by
the Oak Containers C++ toolchain. By using `rules_distroless`, we ensure that
the build-time environment (sysroot) is hermetic, reproducible, and matches the
runtime environment (the system image).

There are two reasons that this is a separate Bazel Module, rather than being
integrated directly into the main Oak repository.

1. Bootstrapping issue: we need this target built in order to properly build the
   llvm_toolchain. But at the time that the toolchains are being built, it's not
   yet possible to build this target.

2. Snapshot availability: historically, we've had issues with the availability
   of the Debian snapshot servers being somewhat unreliable (or, perhaps,
   throttling us). So to reduce CI flakiness, we'll stick to preferring to
   depend on the pre-built sysroot artifacts being available for now.

## Contents

- `sysroot.yaml`: Defines the Debian packages and snapshots to include.
- `sysroot.lock.json`: The lockfile for the Debian packages, generated by
  `rules_distroless`.
- `BUILD`: Contains rules to bundle the sysroot into a tarball.
- `BUILD.sysroot`: A template BUILD file included in the sysroot tarball,
  defining how Bazel toolchains should consume the sysroot contents.

## Workflows

### Updating the Sysroot

If you need to add or update packages in the sysroot:

1. Modify `sysroot.yaml` to add/remove packages or update the snapshot URL.
2. Update the lockfile by running:

   ```bash
   just containers sysroot
   ```

   (This runs `bazel run @sysroot//:lock` under the hood).

3. Commit both `sysroot.yaml` and `sysroot.lock.json`.

### Fast Iteration

For local development and testing of sysroot changes, use:

```bash
just containers sysroot-dev
```

This recipe builds the sysroot without the expensive compression step and prints
the absolute path to the resulting tarball. You can then point the
`oak_cc_toolchain_sysroot` in the root `MODULE.bazel` to this local path for
testing.

## Integration

The sysroot is consumed by the LLVM toolchain defined in the root
`MODULE.bazel`. The `http_archive` named `oak_cc_toolchain_sysroot` downloads
the compressed sysroot tarball, which is then used by `llvm.sysroot`.
